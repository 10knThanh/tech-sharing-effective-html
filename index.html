1 · Vì sao

Vấn đề của diagram truyền thống

Draw.io Figma Excalidraw PowerPoint Mermaid
  • Diagram nhanh lỗi thời
    Project càng phát triển, hình càng xa thực tế — không ai dám tin sơ đồ cũ.
  • Không review được trên Pull Request
    File binary / XML — PR không hiện diff, kiến trúc đổi mà không ai duyệt.
  • Merge conflict không giải được
    Hai người cùng sửa một sơ đồ là phải chọn một bản, vứt bản kia.
  • Khó maintain · không đồng bộ với source code
    Diagram sống ở Drive/Figma, code sống ở Git — hai vòng đời tách rời nhau.

Documentation thường bị bỏ quên sau vài sprint.

2 · Khái niệm

Effective HTML là gì?

Một cách xây dựng diagram bằng HTML + CSS + component — thay vì kéo thả. Node là thẻ div, layout là CSS, diagram là DOM. 

<!-- code bên trái = diagram bên phải -->
<div class="flow">
  <div class="fnode">HTML</div>
  <div class="farrow"></div>
  <div class="fnode">Render</div>
  <div class="farrow"></div>
  <div class="fnode new">Diagram</div>
</div>

6 · So sánh (tiếp) · nói thẳng

Giới hạn của Effective HTML

  • Sơ đồ rất lớn vẫn là điểm yếu
    Bố cục đã có AI lo — với diagram cỡ tài liệu (10–20 node) là quá đủ. Nhưng sơ đồ vài chục node chằng chịt thì AI ước lượng toạ độ, không phải engine auto-routing tối ưu đường nối như Draw.io: mỗi lần chỉnh là một vòng prompt và phải soát lại bằng mắt.
  • Xuất ảnh tĩnh không chuyên
    Cần PNG/PDF để dán vào Confluence hay slide cho non-engineer thì phải in trang hoặc chụp màn hình — Draw.io, Excalidraw làm việc này tốt hơn hẳn.
Cách giảm đau: chia sơ đồ lớn thành nhiều diagram nhỏ theo domain (mỗi diagram một câu chuyện) · việc cần kéo thả nhanh hay xuất ảnh thì Draw.io vẫn ở đó — đúng việc của nó.
7 · Tổng kết

Bốn điều mang về

  • Diagram cũng nên được quản lý như source code.
  • Effective HTML giúp diagram dễ bảo trì, dễ review — và highlight được từng flow khi sơ đồ chồng chéo.
  • AI có thể tạo và cập nhật diagram trong vài giây.
  • Phù hợp với team đang dùng Git và Pull Request — chính là chúng ta.
